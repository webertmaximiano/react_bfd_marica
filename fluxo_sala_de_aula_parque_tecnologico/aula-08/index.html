Aula 08: Introdução ao Git e GitHub

Nesta aula, vamos aprender os conceitos básicos e os principais comandos do Git, o sistema de controle de versão mais popular do mundo, e do GitHub, a maior plataforma de hospedagem de código.

O que é Git?

Git é um sistema de controle de versão distribuído. Mas o que isso significa?

  • Controle de Versão: Ele registra todas as alterações feitas em um arquivo ou conjunto de arquivos ao longo do tempo. Isso permite que você reverta para versões anteriores, compare mudanças e entenda a evolução do seu projeto. É como um "histórico de salvamentos" superpoderoso.
  • Distribuído: Diferente de sistemas centralizados, cada desenvolvedor tem uma cópia completa do histórico do projeto em sua máquina local. Isso torna o trabalho mais rápido e flexível.

1. Configuração Inicial (Feita apenas uma vez)

Antes de começar a usar o Git, você precisa configurar seu nome de usuário e e-mail. Isso é importante porque cada "commit" (salvamento) que você fizer usará essa informação.

git config --global user.name "Seu Nome"
git config --global user.email "seu.email@exemplo.com"

O comando --global faz com que essa configuração seja aplicada a todos os seus projetos Git no seu computador.

2. Comandos Essenciais do Dia a Dia

Estes são os comandos que você usará constantemente.

git init

Para começar a usar o Git em um projeto que ainda não está sob controle de versão, você precisa "inicializar" um repositório. Navegue até a pasta do seu projeto pelo terminal e execute:

git init

Isso cria uma subpasta oculta chamada .git, onde o Git armazena todo o histórico e as configurações do projeto.

O Fluxo Básico: Modify, Stage, Commit

O fluxo de trabalho no Git consiste em três estágios:

  1. Working Directory: Onde você está trabalhando, modificando seus arquivos.
  2. Staging Area (Index): Uma área intermediária onde você prepara ("adiciona") as alterações que deseja salvar.
  3. Repository (.git dir): Onde o Git armazena permanentemente as alterações como um "commit".

git status

Este é o seu comando mais importante. Ele mostra o estado atual do seu repositório: quais arquivos foram modificados, quais estão na staging area e quais não estão sendo rastreados pelo Git.

git status

git add

Para adicionar as alterações de um arquivo (ou vários) à Staging Area, usamos o comando git add.

# Adiciona um arquivo específico
git add nome_do_arquivo.html

# Adiciona todos os arquivos modificados na pasta atual e subpastas
git add .

git commit

Depois que suas alterações estão na Staging Area, você as salva permanentemente no histórico com um "commit". Cada commit tem uma mensagem descritiva.

git commit -m "Sua mensagem descritiva aqui"

Boas práticas para mensagens de commit: Seja claro e conciso. Ex: "Adiciona formulário de contato" ou "Corrige bug no login".

Comandos Avançados: git reset

O comando git reset é usado para desfazer alterações. Ele pode ser um pouco intimidador, pois modifica o histórico de commits, mas é extremamente útil. Vamos ver seus 3 modos principais, do menos ao mais destrutivo.

Para os exemplos, imagine que você quer voltar para o commit anterior. Você pode usar HEAD~1 para se referir a ele.

Modo 1: git reset --soft

Este é o modo menos destrutivo.

  • O que faz? Ele desfaz o último commit, mas mantém suas alterações na "Staging Area".
  • Quando usar? Perfeito para quando você comitou algo, mas esqueceu de adicionar um arquivo, ou quer refazer a mensagem de commit sem perder o que já preparou.
# Volta 1 commit, mas deixa as alterações prontas para um novo commit
git reset --soft HEAD~1

Depois de rodar, se você usar git status, verá os mesmos arquivos de antes na Staging Area, como se você tivesse acabado de usar git add neles.

Modo 2: git reset --mixed (o padrão)

Se você não especificar um modo, o Git usará o --mixed.

  • O que faz? Ele desfaz o último commit e também tira as alterações da Staging Area, colocando-as de volta no seu "Working Directory".
  • Quando usar? Útil quando você quer refazer completamente o último commit, talvez para separar as alterações em commits diferentes ou para descartar algumas delas.
# Volta 1 commit e move as alterações para a sua área de trabalho
git reset --mixed HEAD~1
# Ou simplesmente:
git reset HEAD~1

Depois de rodar, git status mostrará os arquivos como "modificados", mas não prontos para comitar ("not staged").

Modo 3: git reset --hard (CUIDADO!)

Atenção: Este comando é destrutivo e pode causar perda de trabalho. Use com extrema cautela.

  • O que faz? Ele desfaz o último commit e descarta permanentemente todas as alterações que estavam nele. Seus arquivos voltam exatamente para o estado em que estavam no commit anterior.
  • Quando usar? Principalmente para descartar commits que você tem certeza absoluta que não quer mais e cujas alterações são lixo. Nunca use em commits que já foram enviados para um repositório remoto (com git push).
# Volta 1 commit e APAGA TODAS as alterações permanentemente
git reset --hard HEAD~1

Depois de rodar este comando, as alterações dos arquivos são perdidas. Não há um "desfazer" fácil para o reset --hard.

3. O que é o GitHub?

GitHub é uma plataforma online que hospeda repositórios Git. Ele funciona como uma rede social para desenvolvedores e oferece ferramentas para colaboração.

  • É um "backup na nuvem" para o seu código.
  • Permite que você trabalhe em equipe no mesmo projeto.
  • Facilita a revisão de código e a discussão sobre as alterações.

Git é a ferramenta, GitHub é o serviço que usa a ferramenta.

4. Trabalhando com Repositórios Remotos (GitHub)

Para conectar seu repositório local a um repositório no GitHub (que você deve criar primeiro no site do GitHub), usamos os seguintes comandos.

git remote add origin

Este comando conecta sua pasta local a um endereço remoto no GitHub. Você só precisa fazer isso uma vez por projeto.

git remote add origin URL_DO_SEU_REPOSITORIO_NO_GITHUB.git

origin é o nome padrão (um apelido) para o seu repositório remoto principal.

git push

Depois de fazer seus commits locais, você precisa "empurrar" (push) essas alterações para o GitHub para que fiquem salvas online e visíveis para outros.

git push -u origin main

-u origin main é usado na primeira vez para ligar sua branch local `main` com a branch `main` remota. Nas próximas vezes, você pode usar apenas git push.

git pull

Se outra pessoa (ou você, de outro computador) fez alterações no repositório do GitHub, você usa git pull para "puxar" (pull) essas atualizações para a sua máquina local.

git pull origin main

git clone

Se você quer baixar um projeto que já existe no GitHub para a sua máquina, você usa o comando clone. Ele baixa o projeto e já configura a conexão remota.

git clone URL_DO_REPOSITORIO_NO_GITHUB.git

Segurança: Removendo Credenciais Salvas

Quando você usa o Git para se conectar a repositórios remotos como o GitHub, ele geralmente salva suas credenciais (usuário e senha/token) para não ter que pedi-las toda vez. Isso é feito por um "Credential Helper".

No entanto, se você estiver usando um computador público ou compartilhado, é crucial remover suas credenciais ao terminar.

Como o Git Salva as Credenciais?

O Git usa um gerenciador de credenciais específico para cada sistema operacional. No Windows, é o Credential Manager.

Removendo Credenciais no Windows (Duas Formas)

Forma 1: Pelo Painel de Controle (Recomendado)

  1. Abra o Menu Iniciar e procure por "Gerenciador de Credenciais" (ou "Credential Manager").
  2. Na janela que abrir, clique em "Credenciais do Windows".
  3. Procure por uma entrada que comece com git:https://github.com.
  4. Clique nela para expandir e selecione a opção "Remover".

Isso remove sua senha ou token do cofre do Windows de forma segura.

Forma 2: Pela Linha de Comando (Avançado)

O Git para Windows vem com uma ferramenta de linha de comando para gerenciar as credenciais. Você pode usá-la para "apagar" as credenciais de um site específico.

# Este comando abre um prompt para você digitar o host do qual quer apagar a senha
# Você digitaria: host=github.com e pressionaria Enter duas vezes.
git credential-manager-core erase

Na próxima vez que você executar um git pull ou git push, o Git pedirá seu nome de usuário e senha novamente, e você pode simplesmente fechar a janela para não salvar as novas credenciais.
